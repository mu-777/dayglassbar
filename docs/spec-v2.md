# DayGlassBar 要件定義 v2

> 画面の縁に常駐する細いバーで、一日の区間（例: 9:00〜17:00）の残り時間を「グラスの水が減る」ように可視化するアンビエントなデスクトップアプリ。数字や色の変化に頼らず、視界の端で残量を体感することを狙う。

## 1. 概要

| 項目 | 内容 |
| --- | --- |
| 形態 | デスクトップ常駐アプリ（オーバーレイ＋トレイ） |
| 対象OS | **Windows 優先・macOS 両対応**（Linux はスコープ外） |
| 中心思想 | 「促すが、急かさない」 |
| 表示 | 画面の縁の細いバー。残り時間ぶんだけ塗りが縮む |

## 2. 目的・非目的

### 目的
- 一日の作業区間の**残り時間を、視界の端で常時うっすら把握できる**こと。
- 視線を奪わない・操作を要求しないこと（アンビエント）。
- 必要なときだけ（ホバー）、正確な時刻と残り時間を確認できること。

### 非目的
- タスク管理・ポモドーロ・通知によるリマインドではない。
- 厳密な生産性計測ツールではない。
- 締切を「煽る」表現（赤色化・点滅・カウントダウン音）は行わない。

### 減少表示の設計ノート
残量を「塗りの長さ」で表すのは、**損失回避**（減っていくものに人は注意を向けやすい）を穏やかに利用するため。ただし色変化や点滅で危機感を煽ると「急かす」側に倒れるため、**減るのは長さのみ・色は一定**を原則とする。

## 3. 時間モデル

### 3.1 日跨ぎ
- 終了時刻は **24時超表記**（`25:00` = 翌1:00）で表現する。
- 区間は**開始日に帰属**する。深夜0時を越えても、その区間を開始した日の設定として扱う。
- 1区間は **24時間未満**とする。
- 「今いる区間」は、前日→今日の順に走査して判定する（前日の夜から続く区間を取りこぼさないため）。

### 3.2 休憩（グレー区間）
- 区間内に休憩を複数設定でき、**残り側にグレーで表示**される。
- **過ぎた休憩は経過時間と一緒に消える**（残り側にのみ意味を持たせ、「これから休める時間」を示す）。
- 常時グレー表示（経過後も薄く残す）案は未決事項（10章）に保留。

## 4. 機能要件

### 4.1 スケジュール
- **二層構造**: 曜日ごとのデフォルト（weekly）＋特定日の上書き（overrides）。上書きが優先。
- **OFF日**: 曜日・特定日とも「無効」にでき、その日はバーを表示しない。
- **バリデーション**（保存時に検証し、NGなら保存しない）:
  - 時刻形式（`H:MM`〜`47:59`）、開始 < 終了、24時間未満。
  - 休憩は区間内・休憩同士は重ならない。
  - 連続する日の区間同士が重ならない（日跨ぎ考慮、隣接（端が一致）は許可）。
  - 上書き日は実際の前後の日（上書き同士でも）と突き合わせて検証。

### 4.2 表示形状

| 配置 | 軸の向き | 起点(S) | 終点(E) | 減る方向 |
| --- | --- | --- | --- | --- |
| 上/下 | 水平 | 左 | 右 | 右へ縮む |
| 左/右 | 垂直 | 上 | 下 | 下へ縮む |

- 配置は**ワークエリアの縁**（タスクバー／Dock／メニューバーを避ける）。
- 太さ・色・不透明度・ディスプレイを設定可能。マルチディスプレイ対応（指定ディスプレイが外れたらプライマリにフォールバック、復帰したら戻す）。
- DPI / 解像度変更によしなに追従する。

### 4.3 視覚表現
- 残り時間ぶんの塗り（fill）が縮小。色は一定（危機色なし）。
- **下地（track）** の表示ON/OFF・濃さを設定可能（軸全長をうっすら表示）。
- 休憩はグレー（3.2）。
- **目盛り（ticks）** の表示ON/OFF・間隔（分）を設定可能。

### 4.4 挙動（ホバー展開）
- **常時クリックスルー**（マウス入力を素通し）。最前面に常駐するが、展開中も含めて操作の邪魔をしない。
- カーソルがバー上に**一定時間（dwell）滞留**したら、バーが拡幅する。拡幅した幅は**いっぱいまで塗りで埋め**、その上に正確な時刻（開始・終了・現在）と残り時間をラベル表示する（細い帯のまま中央に小さく出すのではなく、広がった面全体がバーになる）。
- カーソルが離れたら元の細いバーに戻る。
- 区間外（`empty`）でホバー展開した場合は「区間外」ラベルに加えて**次の区間の開始**（例: `Next Mon 9:00`）を表示する。
- 設定画面は**トレイメニュー**から開く。バーは展開中もクリックを素通しするため、クリックでは開かない（背後アプリの操作を奪わない）。
- ※「目的レベル要件」。実現手段（forward/ポーリング等）は設計判断（docs/design.md）。

### 4.5 常駐運用
- トレイ常駐（Windows: タスクトレイ / macOS: メニューバー）。メニューから設定・終了。
- トレイの先頭行は**今いる区間**を表示する。前日から続く夜跨ぎ区間（例: 日曜 9:00〜27:00）が稼働中なら、その**開始日の曜日と24時超レンジ**（例: `日曜: 9:00〜27:00`）を表示する。区間外なら当日の設定（`今日: 9:00〜17:00` / `今日: 休み`）にフォールバックする。判定は core の `getActiveDaySummary`（前日→今日の走査で active interval を優先・3.1の帰属と整合）。
- **自動起動**（ログイン時）設定。OS標準の仕組みを使用。
- **スリープ復帰対応**: 復帰時に壁時計から再算出（経過時間の積算をしない）。
- **多重起動防止**: 2つ目の起動は既存インスタンスの設定画面を開く。
- 設定の永続化・エラー通知・認証情報の保管は **OS標準に準拠**。
- 設定変更は**即時反映**（再起動不要）。

### 4.6 カレンダー連携
- 設定時刻（区間）内にある予定の時間帯を、バー上で別色に色付けする（read-only）。通常は色変化のみ・**ホバー時に予定タイトルを表示**（スペース不足は省略可）。
- ユーザー向けは**2プロバイダ。各々に表示 ON/OFF と色**を設定:
  - **Google**: クラウド OAuth（Google Calendar API）。
  - **Outlook**: 接続方法を**いずれか一つ**選ぶ — **ローカル**（Windows＋クラシック Outlook を COM で読み取り・サインイン/承認不要）／**クラウドAPI**（Microsoft Graph OAuth。企業テナントは管理者承認が要る場合あり）。（現状、クラウドAPIは設定UIでは未対応扱い＝選択すると説明表示＋接続ボタン無効。経緯は docs/calendar-integration.md 決定0b）
- 「区間＝1本のバー」の抽象は変えない（予定は区間内のオーバーレイで、新しいバーにはしない）。
- 色付けは休憩と同じ**「残り側のみ」**（過ぎた予定は経過分と一緒に消える）。**終日予定・辞退済み・空き(free)表示の予定は対象外**。
- 認証情報の保管は OS 標準（暗号化）。**OAuth トークンは設定のエクスポートに含めない**（表示設定は非秘匿で含む）。実装方針（承認不要＝ローカル優先・PKCE/ループバック・client_id のみ同梱・依存ゼロ・秘匿分離）は `docs/design.md`「カレンダー連携」・`docs/calendar-integration.md`。
- 新しい Outlook / Web は完全ローカル不可（COM 無し）→ クラウド OAuth（承認のうえ）。**ICS 公開URL購読は一度実装後に撤回**（提供側キャッシュで更新が数時間〜1日遅れ＝予定変更への追従が要件不足。`docs/calendar-integration.md` 決定0）。

## 5. 表示状態

| 状態 | 条件 | 表示 |
| --- | --- | --- |
| 区間内（active） | 現在が区間内 | 下地＋fill（＋休憩グレー・目盛り） |
| 区間外（empty） | 当日有効だが区間の前/後 | 下地のみ（設定に従う） |
| 非表示（hidden） | 当日OFFかつ夜跨ぎ区間もない | ウィンドウごと非表示 |

## 6. 非機能要件
- 常時起動を前提に、過度なCPU使用を避ける（秒更新＋近傍時のみ高頻度ポーリング）。
- カリカリのパフォーマンスチューニングは不要（個人用・趣味の範囲）。
- メモリは Electron 相応（150MB級）を許容。

## 7. 時刻シミュレーション（開発支援）
環境変数で「今」を操作できる:

| 変数 | 例 | 意味 |
| --- | --- | --- |
| `DAYGLASSBAR_FAKE_NOW` | `"2026-06-15 16:30"` | 起点となるローカル時刻 |
| `DAYGLASSBAR_TIME_OFFSET_MIN` | `120` | 現在時刻を分単位でずらす |
| `DAYGLASSBAR_TIME_SCALE` | `60` | 早送り（60=実1秒で60秒進む。0=停止） |

## 9. スコープ外
- Linux 対応、モバイル、Webクライアント。
- タスク管理・通知リマインド・生産性計測。
- 排他フルスクリーンアプリの上への重畳（OS仕様上隠れるのは期待動作）。
