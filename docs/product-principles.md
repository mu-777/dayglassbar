# プロダクト原則（DayGlassBar）

このアプリの開発で、ユーザーが個別に指示しなくても常に守る一般原則。
過去のフィードバックから抽象化したもの。`CLAUDE.md` の「不変条件」（アンビエント性・「促すが急かさない」・クリックスルー）を補完する位置づけ。

新機能・デフォルト値・UI 変更を行うたびに、ユーザーが明示しなくても先回りでこれらを満たすこと。

## 原則

1. **初回起動で「ちゃんと動いている」と必ずわかること。**
   デフォルト設定は、時刻・曜日・実行環境に関係なく必ず目に見える変化を生む状態にする。条件次第で無反応・不可視になるデフォルトを出荷しない（例: スケジュール外の時間帯に起動しても下地・目盛りで存在がわかる／太さは気づける値にする）。

2. **デフォルトは最も無難で影響の小さい選択肢にする。**
   OS のクローム（タスクバー/Dock/メニューバー/ウィンドウバー）や他アプリの一般的 UI と衝突しない配置・値を選ぶ（例: 表示辺は右をデフォルト）。

3. **エンドユーザー向け UI に開発者・内部向けの項目を出さない。**
   デバッグ/開発機能は環境変数など別経路に置き、設定画面など利用者が触る面には出さない。

4. **複数 PC で同じ使い勝手を再現できる手段を用意する。**
   設定のエクスポート/インポート等。クラウド前提にしない、ローカルファイルで十分。

5. **UI の見た目・アフォーダンスは、一般的な慣習よりこのアプリの実挙動に一致させる。**
   慣習と衝突する場合はアプリの実際の挙動を優先する。

6. **主要/確定アクション（保存・適用）や破壊的アクションは、副次的なユーティリティ操作と視覚的に分離・グルーピングする。**
   誤操作を誘発する隣接配置をしない（例: エクスポート/インポートは左寄せにして「保存して適用」から離す）。

7. **時刻・日付などの境界（深夜0時またぎ・日跨ぎ区間）を、素朴なカレンダー日ではなく実際にアクティブな状態に追従させて正しく扱う。**
   表示は「今アクティブな区間」を反映する（例: 日曜 9:00–27:00 が効いている月曜 2:00 は、月曜ではなく日曜の区間を表示する）。

## なぜ

- ユーザーは初回体験での混乱（「起動したのに何も起きない」）を最も嫌う。
- 設定は複数端末で揃えたい。
- UI が実挙動と矛盾していると不信感につながる。
- 主要操作と副次操作の混在は誤操作を生む。
- 時刻アプリでは境界処理の誤りが致命的に体験を損なう。
